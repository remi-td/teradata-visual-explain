⚠ PRODUCT JOIN: A Cartesian-style join between Spool 16 (27,849 rows) and Spool 19 (20 rows).

Why Product Join? Condition only applies to non-matching rows: "(FieldType IN ('A1','AN')) AND (UDTypeId NOT NULL)". Optimizer must cross-scan Spool 19 (tiny) against Spool 16 to resolve array type names.

Left outer: All 27,849 rows from Spool 16 are preserved; Spool 19 enriches only array-type columns.

Risk: If Spool 19 grows significantly, this becomes a serious bottleneck. Currently Spool 19 = 20 rows, keeping it manageable.

Why it exists: COLUMNSV resolves array element type names (UDT arrays) — requires this conditional lookup pattern.

Can it be fixed? Not without rewriting DBC.COLUMNSV view definition — this is inherent to the system view design.

Optimization Recommendations
🔴 HIGH PRIORITY: Avoid SELECT * — Filter by Database and Columns
Impact: MajorEffort: Low

SELECT * FROM DBC.COLUMNSV retrieves all columns across all 27,851 column definitions in the data dictionary — resulting in a 597 MB spool. In most cases you only need columns for a specific database or table.

Add a WHERE filter and select only needed columns: 
-- Instead of this:
SELECT * FROM DBC.COLUMNSV;

-- Do this (filter by database):
SELECT DatabaseName, TableName, ColumnName, ColumnType, 
       ColumnLength, Nullable, DefaultValue
FROM DBC.COLUMNSV
WHERE DatabaseName = 'YOUR_DATABASE_NAME'
ORDER BY TableName, ColumnId;
Expected impact: 80–95% reduction in spool size and execution time depending on database size.
🔴 HIGH PRIORITY: Understand the 2 Product Joins (Steps 10 & 13)
Steps: 10, 13Impact: Architecture

Two product joins exist in this plan — these are by design within the DBC.COLUMNSV view definition. They exist to conditionally resolve UDT array element type names only for columns with FieldType IN ('A1','AN').

Why product joins? The join condition only applies to a subset of rows, requiring a Cartesian scan of the lookup table (Spool 19 = 20 rows, Spool 23 = 76 rows) against the main dataset. Since both lookup tables are tiny, the product join is currently manageable.

Risk: If your system has many UDT types or array columns, these steps will grow. Monitor if execution time increases.

Workaround: If you don't need UDT/array column type resolution, consider using DBC.COLUMNS (base table) instead of DBC.COLUMNSV: 
-- Use base table for simpler queries (no product joins):
SELECT DatabaseName, TableName, ColumnName, ColumnType
FROM DBC.COLUMNS
WHERE DatabaseName = 'YOUR_DB';
⚠ MEDIUM PRIORITY: group_amps in Step 13 — Potential Skew
Step: 13Impact: Moderate

The final result (Spool 1) is built in (group_amps) mode — meaning only a subset of AMPs holds the final data. This can cause uneven AMP load if those AMPs are already busy.

This is a consequence of the product join — the optimizer builds locally on AMPs that have data, rather than redistributing. Monitor AMP skew if this query runs frequently or concurrently.

Mitigation: Adding a WHERE filter (recommendation #1) will drastically reduce the result set and AMP pressure.
⚠ MEDIUM PRIORITY: 8 Steps with No-Confidence Estimates
Steps: 3.3, 4, 5, 6, 7, 10, 11, 13Impact: Plan Quality

Most join steps show No Confidence — the optimizer cannot reliably estimate intermediate row counts. This means the execution plan may not be optimal for your actual data.

Why? DBC.* system tables are internal — you cannot directly collect user-level statistics on them. The no-confidence ratings are expected for system dictionary views.

Impact: Currently limited because the 2.64s estimate is reasonable. The no-confidence cascades from DBC.OU (Step 3.3) having no stats.

Action: No direct user action possible for DBC stats. Focus on adding WHERE predicates to reduce working data sets.
💡 LOW PRIORITY: Consider DBC.COLUMNS vs DBC.COLUMNSV
Impact: MinorEffort: Low

DBC.COLUMNSV is a view that enriches DBC.COLUMNS with human-readable names (database names, table names, UDT type names). This requires 13 steps and ~2.64s even without data filtering.

DBC.COLUMNS is the underlying base table — faster access for analytical queries that don't need the enriched name columns.

Use COLUMNSV when you need: TableName, DatabaseName, ColumnType (as text), CreatorName, LastAlterName.
Use DBC.COLUMNS when you need: Raw IDs (TableId, DatabaseId) for joining to your own metadata.
